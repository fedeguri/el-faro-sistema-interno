research. Sonnet verificó las afirmaciones técnicas/legales contra fuentes reales antes de aprobar,
y empaquetó el resultado en una skill.

## ACELERADOR DE SHOPIFY (SOP POR CONECTOR)

Documento completo: `ACELERADOR_SHOPIFY_SOP.md`. Resumen operativo:

**Modelo:** una tienda conectada por vez. `get-shop-info` es siempre el Paso 0 — ninguna escritura
sin confirmar contra qué tienda se está trabajando.

**5 hallazgos verificados contra las herramientas reales del conector** (no de memoria):
1. `create-product` crea en DRAFT por defecto — cargar todo el catálogo en borrador, verificar,
   activar en bloque al final con `bulk-update-product-status`.
2. Las colecciones smart aceptan reglas por TAG — arquitectura de 3 niveles de ticket con tags
   `nivel-1/2/3`, las colecciones se pueblan solas.
3. **No existe herramienta `upload-image` en el conector** (corrección de Sonnet sobre el diseño
   original de Fable) — las fotos siempre se hostean antes en Netlify/repo, nunca vía el conector.
4. `run-analytics-query` (ShopifyQL) consulta `conversion_rate` directo — el corte de Semana 4 se
   verifica sin entrar al admin.
5. `create-discount` exige elegibilidad y fecha explícitas — nunca asume valores por defecto.

**Decisión de flujo:** generar TODO el copy primero (vía `el-faro-reescritor-descripciones`), cargar
productos ya completos. Nunca cargar "en blanco" y actualizar después.

**Estado:** diseño completo, corregido y verificado. Pendiente de prueba en vivo — candidato: Shudita.

## BIBLIOTECA DE INTERACCIONES PREMIUM

Documento completo: `BIBLIOTECA_INTERACCIONES_PREMIUM.md`. Módulo de código real: `capa-premium.html`
(empaquetado como skill `el-faro-capa-premium`) — transversal a las 6 verticales, no trae paleta
propia, lee las variables CSS del sitio anfitrión.

**9 patrones activos, todos medidos y testeados:** validación que enseña, confirmación con peso
<300ms, botón magnético, tilt 3D, resplandor de cursor, cursor linterna, sticky reveal, texto
escalonado, y **hero rotativo con zoom sutil (Ken Burns, hasta 5 imágenes)** — este último
generalizado desde el hero de Turismo a pedido de Fede. Ningún patrón supera 3,15KB; módulo
completo 17KB, cero bloqueo de render.

**Regla de banco de imágenes (Paso 1b de `el-faro-perfiles-cliente`):** si no hay fotos reales del
cliente para el preview, se completa con banco de imágenes gratuito (Unsplash/Pexels/Pixabay) —
nunca vacío, nunca IA (no es gratis todavía). Excepción dura: fotos de una persona específica
(Profesionales, Salud, Venta Directa) nunca se reemplazan por un modelo de banco genérico — ahí se
deja el placeholder marcado.

**Regla de uso:** ningún patrón de esta biblioteca reemplaza la firma visual de una vertical — se
suma como capa de interacción opcional (paquete Full), nunca la sustituye. Salud y Profesionales se
mantienen con movimiento mínimo por diseño; Turismo, Estética, Inmobiliarias y Venta Directa tienen
más margen.

**Cómo crece esta biblioteca:** (1) lo que funciona con un cliente real se generaliza y se suma acá;
(2) revisión de tendencias cada 2-3 meses, no más seguido; (3) ganchos `data-variante` ya cableados
al GA4 de cada sitio — cuando haya datos de 6-8 clientes, medir qué patrones correlacionan con
conversión real, no solo con verse bien.

ejecutar vía conectores, y documentar. No gastar horas de Fable en trabajo mecánico.

## PROTOCOLO DE SEGURIDAD Y CONTINUIDAD DEL SISTEMA

Agregado en v4.0, a pedido explícito de Fede — para que nada se pierda si algo se rompe.

### Jerarquía de fuente de verdad
1. **Estos archivos del Project** (`EL_FARO_ARCHIVO_MAESTRO_v4.md` + compañeros) — nunca dependen
   de una conexión externa, se leen siempre.
2. **Notion** — capa operativa, búsqueda, tableros, bitácora del día a día. Se sincroniza DESDE los
   archivos, no al revés.
3. **Las skills instaladas** — versión ejecutable de los archivos de vertical. Si una skill falla o
   desaparece, se puede reconstruir leyendo el archivo de vertical correspondiente.

### Regla de sincronización
Después de cualquier cambio real al sistema (vertical nueva, corrección de un hallazgo técnico,
skill actualizada), **actualizar Notion y el archivo del Project en la misma sesión.** No queda
"para después" — es lo que pasó hoy con esta misma sesión.

### Cómo se corrigen los errores (sin romper lo que ya funciona)
- Nunca sobrescribir silenciosamente. Todo cambio de versión anota qué cambió y por qué en la
  sección de changelog al final de este archivo — mismo criterio que ya se usaba en v2.0→v3.0.
- Antes de corregir una afirmación técnica o legal, verificarla contra la fuente real (herramienta,
  ley, documentación) — no confiar en la memoria de ninguna sesión anterior. Así se detectó y
  corrigió el error de `upload-image` en el Acelerador de Shopify.
- Un hallazgo nuevo primero se registra (Bitácora/archivo), después se propaga a las skills — nunca
  al revés. Si algo se aplica antes de estar bien documentado, se pierde el porqué cuando haga falta
  revisarlo.
- Ante la duda entre "arreglar rápido" y "arreglar bien", en el sistema (no en la entrega de un
  cliente puntual) se prioriza arreglar bien — este archivo es la base de los próximos años de
  El Faro, no un entregable con fecha de vencimiento.

### Redundancia recomendada (fuera de Claude)
Los archivos de este sistema y las 5 skills `.skill` conviene respaldarlos también fuera del
Project de Claude — por ejemplo, en el mismo repositorio de GitHub donde ya vive el código de los
clientes, o en una carpeta de Drive. Es el mismo principio de ownership que El Faro ya aplica al
código de los clientes, aplicado a su propio sistema interno.
